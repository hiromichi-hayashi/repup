# Spec-Driven Development

設計書に書かれた事実に基づいて実装を進めるスキル。推測や思い込みによる実装を防ぎ、不明点は必ずユーザーに確認する。

## ワークフロー

### Phase 0: タスク開始

1. Linear Issue ID を受け取ったら、Linear MCP ツール (`get_issue`) で Issue の内容を取得する
2. Issue のタイトル・説明から「何をするのか」を正確に把握する
3. 現在のブランチをそのまま作業ブランチとして使用する（ブランチの作成・チェックアウトは行わない）
4. 現在のブランチ名に Issue ID（例: `DEV-12`）が含まれているか確認する
   - 含まれている場合: そのまま作業を開始する
   - 含まれていない場合: AskUserQuestion で「現在のブランチ `<branch>` は Issue `<ID>` と一致しませんが、このまま続行しますか？」と確認する

### Phase 1: 設計書読み込みとブレスト

1. [doc-map.md](references/doc-map.md) を参照し、Linear Issue の内容に関連する設計書セクションを読み込む。大きなドキュメントは grep パターンで該当箇所を特定してから読む。

2. 読み込んだ設計情報を箇条書きで整理する（内部メモとして）:
   - DB: 関連テーブル、フィールド、リレーション
   - API: エンドポイント、リクエスト/レスポンス型
   - UI: 画面レイアウト、操作フロー
   - 制約: 受入基準、非機能要件

3. **`superpowers:brainstorming` スキルを発動**し、Linear Issue の内容と設計書の事実をインプットとしてユーザーと壁打ちする。ここで実装方針を決定する。

### Phase 2: 不整合・不明点の検出と確認

読み込んだ設計情報に対して以下をチェックする:

**即座に AskUserQuestion で確認すべきケース:**
- 設計書に記載がない仕様判断が必要な場合
- 設計書間で矛盾がある場合（例: RD-001 のデータ項目と DD-001 のスキーマが不一致）
- 実装計画の手順と現在のコードベースの状態に齟齬がある場合
- 設計書の記載が曖昧で、複数の解釈が可能な場合

**自己判断してよいケース:**
- 設計書に明記されている仕様をそのまま実装する場合
- 設計書が言及しない純粋な実装詳細（変数名、ファイル分割など）

設計書変更タスクの場合は [consistency-checks.md](references/consistency-checks.md) を使って影響範囲を確認し、不整合があれば報告する。

### Phase 3: 実装計画の作成

**`superpowers:writing-plans` スキルを発動**し、Phase 1-2 で整理した設計情報とブレスト結果をもとに実装計画を作成する。

- 出力先: `docs/plans/` ディレクトリ
- 設計書の記載を根拠とし、引用時はドキュメントID とセクション番号を明記する（例: 「DD-001 3.2 の articles テーブル定義に従い…」）

### Phase 4: 実装

**`superpowers:executing-plans` スキルを発動**し、Phase 3 で作成した計画に従って実装を進める。

実装中の原則:
- 設計書の記載と計画を根拠として実装する
- 実装中に新たな不明点が出たら、推測で進めずに AskUserQuestion で確認する
- 設計書に書かれていない実装判断を行った場合、その旨をユーザーに伝える

## 禁止事項

- 設計書を読まずにコードを書き始めること
- 設計書の記載と異なる仕様を暗黙的に実装すること
- 不明点を推測で埋めて実装を進めること
- 設計書に書かれている情報をユーザーに再質問すること（元の設計書を読み返す）
- Linear Issue を確認せずに実装を始めること
