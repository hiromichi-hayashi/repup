## 1. 全体像

### 1.1 システム構成

オンラインファースト。サーバーが真実のソース(SSoT)。端末はローカル DB を持たない。

```
[iOS / React Native + Expo]
   │  HTTPS (JSON / REST)
   ▼
[API: Hono (TypeScript)]
   │  Prisma
   ▼
[DB: Neon (PostgreSQL)]

[cron] ──▶ リマインド/アラート判定 ──▶ [Expo Push] ──▶ 端末
```

技術選定の根拠と棄却理由は [05 技術スタック](05_tech_stack.html)、通知の配信設計は [06 通知設計](06_notification_design.html) を参照。

### 1.2 API 規約

- 認証は Bearer トークン(Better Auth セッション)。`Authorization: Bearer <token>`
- 日時は **内部 UTC / 表示 JST**。API は ISO8601(UTC)で送受信
- スキーマは zod で定義し、リクエスト/レスポンスを検証
- エラーは `{ "error": { "code": string, "message": string } }` 形式、HTTP ステータスを併用

## 2. BASE-01 認証・アカウント

### 2.1 要件

- ログインは次の **2 方式のいずれか**(どちらで入っても同一アカウントに紐づく)
  - **Sign in with Apple**
  - **メール + パスワード**
- メール+パスワードは、**パスワードリセット(メールによる再設定フロー)** とメール確認を含む
- 認証基盤は **Better Auth**(Hono + Prisma 統合)。**パスワードはハッシュ化して保存**(平文保持しない)
- 初回サインイン時に User レコードを作成し、初回設定(目標設定 / SC-02)へ誘導

### 2.2 エンドポイント

| メソッド | パス | 説明 |
|---|---|---|
| `*` | `/auth/*` | Better Auth が提供(Apple サインイン・メール+パスワードのサインアップ/サインイン・パスワードリセット・セッション) |
| GET | `/account` | 現在のユーザー情報 + 初回設定済みフラグ(詳細は §3) |

## 3. BASE-02 アカウント / 目標設定

実体は **アカウント情報** と **個人の目標設定** の2つ。いずれも**個人単位**で、チームは関与しない(チームは F06)。設定値は設定画面(SC-11)からいつでも編集できる。

### 3.1 アカウント情報

- **表示名**
- **メール**(認証由来。BASE-01)
- **Push トークン**(Expo Push、端末ごと)

### 3.2 目標設定(個人)

- **目的 / 動機**(任意, `purpose`): 「なぜ行くか」の自由記述。例「継続したい」
- **通う日**: いつ通うか。**F01 スケジュール(繰り返し)に一本化**する(§4)。「どれだけ・いつ行くか」はこの通う日が表し、連続カウント(§7.1)の基準にもなる
- **既定の時間帯**(`defaultTimeOfDay`): 朝 / 昼 / 夜。新規スケジュールの初期値(朝を既定として提案。強制しない。[01](01_concept.html) §4.5)

> **頻度は「週 N 回」の数値で持たない**(`weeklyTarget` は廃止)。通う日をスケジュールで具体的に定義し、頻度の二重管理を避ける。目的(`purpose`)と通う日(スケジュール)は別概念。

アカウント情報(`purpose` / `defaultTimeOfDay` 等)は **`/account`** で、通う日は **`/schedules`**(§4.3)で扱う。

### 3.3 「オンボーディング」の位置づけ

「オンボーディング」は **独立した API ではなく、初回起動時に上記設定を一度通す UI フロー**(画面 SC-02)の呼び名にすぎない。完了は**初回の保存(`PATCH /account`)時にサーバーが `onboardedAt` を記録**して表現する(専用の完了エンドポイントは持たない)。

### 3.4 エンドポイント

| メソッド | パス | 説明 |
|---|---|---|
| GET | `/account` | アカウント情報 + 目標設定 + 初回設定済みフラグ(`onboardedAt`) |
| PATCH | `/account` | アカウント情報・目標設定の更新(表示名・目的・既定時間帯)。初回保存時に `onboardedAt` を記録。通う日は `/schedules`(§4.3) |
| POST | `/account/push-token` | Expo Push トークンを登録/更新 |

## 4. F01 スケジュール & リマインド

### 4.1 要件

「**通う日**」を事前に決め、当日(と前日)にプッシュで背中を押す。**「いかなきゃ」を作る中核**であり、同時に**連続カウント(§7.1)の判定基準**でもある(通う日 = スケジュールに一本化)。

- 繰り返しパターンに対応:
  - **毎週**: 指定曜日(例 月・水・金)
  - **N週ごと**: 指定曜日(例 2週に1回 月曜)
  - **毎月**: 指定日(例 毎月1日・15日)
  - **単発**: 特定日
- 予定ごとに時間帯(開始予定時刻)を持つ。朝を既定として提案([01](01_concept.html) §4.5)
- 当日リマインド + 前日リマインド(チェックリスト未達アラートと連動、§F02)

> 初回の通う日はオンボーディング(SC-02)で設定する。連続判定がこれに依存するため、繰り返しスケジュールは早期(P1)に用意する([07](07_implementation_roadmap.html))。

### 4.2 データと予定日(オカレンス)

`Schedule`(詳細は [03 DB設計](03_database_design.html) §Schedule)。繰り返しは `recurrenceRule`(JSON)+ `startDate`(N週ごとの基準週)で表現する。

`recurrenceRule` の型(曜日は ISO: 1=月 … 7=日):

```
単発:     { "type": "single",  "date": "2026-06-15" }
毎週:     { "type": "weekly",  "interval": 1, "weekdays": [1,3,5] }
2週に1回: { "type": "weekly",  "interval": 2, "weekdays": [1] }
毎月:     { "type": "monthly", "days": [1,15] }
```

スケジュールは指定期間に対して**予定日(オカレンス)の集合**を生成する。リマインド配信(§4.4)も連続判定(§7.1)も、この予定日を基準にする。複数スケジュールがある場合、同一日付の予定日はまとめて 1 つの「通う日」として扱う。

### 4.3 エンドポイント

| メソッド | パス | 説明 |
|---|---|---|
| GET | `/schedules` | 自分の予定一覧(期間指定可) |
| POST | `/schedules` | 予定作成(単発 or 繰り返し) |
| PATCH | `/schedules/:id` | 予定変更 |
| DELETE | `/schedules/:id` | 予定削除 |

### 4.4 リマインド配信

cron が定期実行し、対象予定を抽出して Expo Push を送る。配信タイミング・ペイロード・冪等性は [06 通知設計](06_notification_design.html) に定義。

## 5. F02 準備チェックリスト

### 5.1 要件

行く前の段取りをアプリが持つ。「準備がだるい / 忘れる」への対策。

- ユーザーは**チェック項目テンプレート**を自由に編集(例: タオル、シューズ、**コワーキングスペース予約済み?**)
- 予定(Schedule)に紐づくチェックリストを生成し、各項目を ON/OFF
- **前日**時点で未達(未チェック項目あり)なら**アラート**を出す(§06)

### 5.2 データモデルの考え方

- `ChecklistTemplateItem`: ユーザーごとの「項目の定義」(再利用される)
- `Checklist` + `ChecklistItem`: 特定の予定に対する実体(チェック状態を持つ)

予定を作ると、テンプレートから当該予定用のチェックリストが複製される。テンプレートを後で変えても過去の予定の実体は変わらない。

### 5.3 エンドポイント

| メソッド | パス | 説明 |
|---|---|---|
| GET | `/checklist-templates` | テンプレート項目一覧 |
| PUT | `/checklist-templates` | テンプレート項目の一括更新 |
| GET | `/schedules/:id/checklist` | 予定のチェックリスト取得 |
| PATCH | `/checklists/:id/items/:itemId` | 項目のチェック状態更新 |

## 6. F03 記録(行った証明)

### 6.1 要件

MVP の「行った証明」は**自己申告**。記録の完了をもって「行った」とする。

### 6.2 状態遷移

予定日(または任意の日)ごとに、その日の状態は次のいずれか。

```
            ┌──────────┐
            │  BLANK   │  予定日が来た / 任意日。まだ何も確定していない
            └────┬─────┘
       ┌─────────┴──────────┐
       ▼                    ▼
 「記録する」           「行かなかった」
       ▼                    ▼
 ┌──────────┐         ┌──────────┐
 │ RECORDED │         │  SKIPPED │
 │ =行った   │         │ =サボり   │
 └──────────┘         └──────────┘
```

- **BLANK**: 確定前。予定があったが未操作の状態。後から RECORDED / SKIPPED のどちらかに確定できる
- **RECORDED**: 記録(内容入力)を完了 → **「行った」**。連続カウント・トロフィーに加算(§F04)
- **SKIPPED**: 「行かなかった」を選択 → サボり。連続カウントを途切れさせ、降格評価の対象(§7.4)

> BLANK のまま放置された過去日は「未確定」。連続カウントの判定では、確定するまで連続を伸ばさない(RECORDED で初めて加算)。降格の評価対象にするかは §7.4 で定義。

### 6.3 記録の内容(MVP)

- 必須: 実施日
- 任意: メモ、種目/部位の簡易タグ、所要時間。**MVP では軽量**に保ち、習慣化の可視化を優先

### 6.4 エンドポイント

| メソッド | パス | 説明 |
|---|---|---|
| GET | `/records` | 自分の記録一覧(期間指定可) |
| POST | `/records` | 記録を作成(= RECORDED に確定) |
| POST | `/records/skip` | 指定日を SKIPPED に確定 |
| PATCH | `/records/:id` | 記録の編集 |
| DELETE | `/records/:id` | 記録の取り消し(RECORDED → BLANK へ戻す) |

## 7. F04 報酬・ペナルティ(積み重ねの増減)

記録(F03)に応じて積み重ねが**増える(報酬)/減る(ペナルティ)**。両者は同じ「積み重ね」システムの増減方向であり、**ペナルティは独立機能(旧 F05)としては持たず本機能に内包する**。

### 7.1 連続カウント(ストリーク)

**予定日(スケジュールのオカレンス, §4.2)を守れている連続回数**。決めた通う日を皆勤で通えている限り伸び、**経過した予定日を 1 つでも落とすと 0 に戻る**(=皆勤の連続)。毎日前提ではなく、各自が決めた通う日が基準。

#### 予定日の達成判定

評価日を `today` とする。予定日 `D` の状態は:

- **達成**: `D` に RECORDED がある
- **未達**: `D` に RECORDED が無く、かつ `D` の日が終了済み(`D < today`)または SKIPPED 確定
- **保留**: `D` に RECORDED が無く、`D == today`(進行中)。連続には数えず、壊しもしない

> **予定外の日(オカレンス外)の RECORDED は連続に作用しない**(伸ばさず・壊さず=ボーナス記録)。判定対象は予定日のみ。

#### 算出(canonical)

連続は記録履歴とスケジュールから決まる**導出値**。`UserStats.currentStreak / longestStreak` にキャッシュし、**Record の確定/取消時**と**日次 cron**(経過した予定日の未達反映)で再計算する。

1. 自分の予定日を新しい順に走査する
2. **保留**(今日の未記録)はスキップする
3. 最新側で最初に当たる**判定確定**の予定日が **未達** なら `currentStreak = 0`
4. **達成**が続く限り 1 ずつ加算し、**未達**に当たったら停止
5. `longestStreak = max(longestStreak, currentStreak)`

> 1 日 1 確定([03](03_database_design.html) §Record の `(userId, date)` 一意)。過去のブランク予定日を後から RECORDED にすれば、その予定日は達成に変わり連続が再計算される。

#### 表示(SC-01)

- `currentStreak ≥ 1`: 「○回連続いけてる!💪」
- `currentStreak = 0`: 連続表示は出さず、「今日から始めよう」等の起点コピー

#### 例(毎週 月・水・金)

- 月・水・金と通えている → **連続 3**
- 月・水は通い、金を落とした(その日終了まで RECORDED なし)→ 金で途切れ **連続 0**、次に通った予定日から **連続 1**
- 予定日でない火曜に行って記録 → 連続は変わらず(ボーナス)

### 7.2 トロフィー

累積記録数・連続記録に応じて **ブロンズ / シルバー / ゴールド** を付与する。

| ランク | 例(累積/連続の目安) |
|---|---|
| ブロンズ | 初回〜継続の入口 |
| シルバー | 一定の累積 + 連続を達成 |
| ゴールド | 高い累積 + 長い連続を維持 |

> 付与の具体閾値は実装時に確定し、表を更新する。降格(ペナルティ)は §7.4。

### 7.3 成長グラフ

- 記録回数の推移(週次/月次)を可視化し、「自分の成長具合が一目で見える」を満たす

### 7.4 ペナルティ(積み重ねの減少)

サボりに痛みを持たせる。ペナルティは**個人のトロフィー降格に閉じる**(チームへの通知は持たない / 対象外は [01](01_concept.html) §7.3)。

- **連続カウント途切れ**: 予定日の**未達**(その日終了まで RECORDED なし、または SKIPPED)で連続が 0 に戻る(算出は §7.1)
- **トロフィー降格**: サボりが続くと現在ランクが 1 段下がる(ゴールド→シルバー→ブロンズ)。**cron による定期評価**(予定日の未達が続く)で判定。具体条件は実装時に確定し本節を更新

### 7.5 エンドポイント

| メソッド | パス | 説明 |
|---|---|---|
| GET | `/account/stats` | 連続カウント・累積回数・現在ランク・推移データ |
| GET | `/trophies` | 保有トロフィー一覧と次ランクまでの条件 |

## 8. F06 チーム

### 8.1 要件

一人にしない。チームで進捗を共有し、**チーム内ランキング**で励む。

- チーム作成(作成者がオーナー)、招待コード or リンクで参加
- メンバーの**進捗が見える**(直近の記録状況・連続カウント)
- **チーム内ランキング**(連続カウント or 期間内記録数で順位付け)
- ランキングは**チーム内のみ**(対象外は [01](01_concept.html) §7.3)

### 8.2 プライバシー

- 共有するのは記録の有無・回数・連続など**進捗系のみ**。記録メモ等の詳細は共有しない(MVP)

### 8.3 エンドポイント

| メソッド | パス | 説明 |
|---|---|---|
| POST | `/teams` | チーム作成 |
| GET | `/teams/:id` | チーム情報・メンバー進捗 |
| POST | `/teams/:id/join` | 招待コードで参加 |
| DELETE | `/teams/:id/members/:userId` | 脱退/除名 |
| GET | `/teams/:id/ranking` | チーム内ランキング(指標・期間指定) |
