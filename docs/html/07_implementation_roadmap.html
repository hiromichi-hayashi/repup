DOCUMENT 07

実装ロードマップ

バージョン: 1.0 作成日: 2026年6月 スコープ: 実装順序・依存関係・段階移行

機能 ID は 02 機能設計 に対応。

1. フェーズ全体像

[BASE 環境構築]  ← 環境構築 + 全テーブル作成 + 認証 を“ここで完了させる”
      │
      ▼  以降は機能ごとに API + フロントを積む
[P1 アカウント] → [P2 記録の核] → [P3 段取り] → [P4 報酬] → [P5 チーム] → [P6 仕上げ]
                                                                        → [2nd 以降]
  • BASE で土台を作り切る: 環境構築・全テーブルの作成・認証(BASE-01)。以降スキーマ変更は原則発生させない
  • P1 以降は機能単位で、各フェーズが API + フロントの両方を完成させる
  • GPS / 全国ランキング / 育成マッチョ / 外部連携は 2nd 以降(01 §7.3)

2. スプリント計画

BASE — 環境構築(土台を完成させる)

目的: この段階で 環境構築・全テーブル・認証 を終わらせ、P1 以降は機能の API+フロントに集中できる状態にする。

項目 内容 対応
環境構築 モノレポ(mobile=Expo / api=Hono / 共有 zod スキーマ)、lint・format、CI、ローカル起動、EAS ビルド・Expo Push の初期設定 —
全テーブル作成 Neon + Prisma(driver adapter)で 03 の全テーブルを一括マイグレーション(User / PushToken / Schedule / Record / UserStats / Trophy / Team / Membership / NotificationLog + Better Auth テーブル) 03
認証(BASE-01) Better Auth(Sign in with Apple / メール+パスワード)・セッション・パスワードリセット BASE-01

BASE 完了の定義(DoD): DB スキーマ全量が適用済み / 認証が一通り動く(Apple・メール+パスワード・リセット)/ mobile・api 双方がローカル起動し疎通する。 前提: スキーマを作り切るため、BASE 着手前に 03 を確定しておく(ロジック系の未確定事項はスキーマに影響しないため後続で可)。

P1 — アカウント / 目標設定 / 通う日(API + フロント)

項目 内容 対応
アカウント / 目標 API /account(アカウント情報+目標設定)・/account/push-token BASE-02
通う日(繰り返しスケジュール) /schedules、recurrenceRule(weekly+interval / monthly / single)、予定日(オカレンス)生成 F01 / SC-04
サインイン/初回設定画面 認証フローと初回設定(目的・通う日・既定時間帯・通知許可)。専用UIは作らず設定の目標設定フォームを流用 SC-03 / SC-02

連続カウント(F04)が予定日に依存するため、通う日(繰り返しスケジュール)の作成・予定日生成は P1 で用意する。リマインド配信は P3。

P2 — 記録の核(最優先の価値 / API + フロント)

項目 内容 対応
記録 API RECORDED / SKIPPED / BLANK 状態遷移 F03
ホーム 今日の状態・記録ボタン・連続カウント表示(暫定) SC-01
記録入力 記録する / 行かなかった SC-05

P2 完了時点で「記録して積み上がる」最小ループが動く。

P3 — 段取り(行かせる仕組み / API + フロント)

項目 内容 対応
通知基盤 cron + Expo Push、N-01・N-02(リマインド)、冪等性。P1 の予定日(オカレンス)を基準に配信 06

通う日(スケジュール)自体は P1 で構築済み。P3 はその予定日に対するリマインド配信を載せる。

P4 — 報酬・ペナルティ(続ける動機 / API + フロント)

項目 内容 対応
連続カウント確定ロジック 予定日(P1) × 記録(P2)で判定。日次 cron で未達反映 F04 §6.1
トロフィー(シーズン制ランク) 6段階・昇格のみ・seasonBestStreak から導出。年次リセット cron F04 §6.2
成長画面 グラフ・現在ランク・次ランクまで・シーズン残り SC-06

P5 — チーム(API + フロント)

項目 内容 対応
チーム CRUD 作成/参加(招待コード) F06 / SC-07
チーム目標 スローガン・共通の通う日(推奨)+ 取り込み(adopt) F06 §7.2 / SC-07
進捗共有 メンバーの連続・直近記録 F06
チーム内ランキング 指標・期間 F06 / SC-08

P6 — 仕上げ

項目 内容
設定画面 通知時刻・ON/OFF・サインアウト(SC-09)
空/異常表示 オフライン・記録再送・空状態
計測 成功指標(01 §8)の計測埋め込み
ベータ配信 TestFlight

3. 依存グラフ

BASE 環境構築(全テーブル + 認証 BASE-01)
   └─ P1 アカウント/目標設定 + 通う日(F01 繰り返しスケジュール → 予定日生成)
         ├─ P2 F03 記録 ─────────┐
         │                       └─ P4 F04 報酬・ペナルティ(予定日 × 記録で連続判定)
         ├─ P3 06 通知   ※リマインドは P1 の予定日が基準
         └─ P5 F06 チーム(F03 の記録に依存)
  • テーブルは BASE で全量作成済み。各機能フェーズは新規マイグレーションを基本伴わず、API+フロントの実装に集中する
  • 連続カウント(F04)は「予定日(P1 スケジュール)× 記録(P2)」に依存。通う日は P1 で先に用意する
  • F03(記録)が中心依存。報酬・チーム・ランキングは記録データに依存する
  • 通知(06)は P1 の予定日の後

4. 段階移行(今回 → 2nd)

2nd 機能 前提・移行時の論点
GPS チェックイン 位置許可・判定精度。F03 の「証明」を自己申告から拡張
全国ランキング「日本で○位」 一定のユーザー母数。集計バッチの追加
育成マッチョ 記録に連動するキャラ育成。記録データを流用
LINE / カレンダー連携 OAuth / Messaging API。通知経路の追加(06 を拡張)

2nd の各機能は、それぞれ独立した spec → plan → 実装サイクルで進める。

5. 開発規律

  • 設計書を SSoT とし、実装が設計と食い違ったらコードを直す前に設計書を直す
  • 未確定事項(02 §8 / 05 §3 / 06 §8)は、着手前に確定して該当節を更新する
  • main に直 push しない / git hook を skip しない
