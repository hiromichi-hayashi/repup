DOCUMENT 01

コンセプト / 要件定義

バージョン: 1.0 作成日: 2026年6月 スコープ: MVP のコンセプト・課題・ゴール・機能スコープの定義

RepUp — 筋トレの「Rep(回数)」を積み「Up」する。行かなきゃの気持ちを仕組みで作り、朝ジム習慣へのシフトを支えるアプリ。

1. このドキュメントの位置づけ

本書は RepUp の Single Source of Truth(SSoT)の起点である。なぜ作るのか・誰の何を解決するのか・MVP に何を入れて何を入れないかを定義する。以降の機能設計(02)・DB設計(03)・画面設計(04)はすべて本書のスコープに従う。

実装と設計が食い違ったら、コードを直す前に設計書を直す。

2. 背景 — As-Is(現状の課題)

オーナー(初期ユーザー像)の生の声を起点にする。

  • 筋トレにお金を払っているのに行っていない(ジム会費が無駄になっている)
  • 夜に行くのが暗黙のルールになっていて、行くと帰りが遅くなるのが嫌
  • そもそも行くのがめんどう。ジムが遠い / 歩きたくない / 準備がだるい
  • 時間がない(本当はなくはない) という言い訳をしてしまう
  • もはや行くこと自体を忘れている(習慣から外れている)
  • 「筋トレ楽しい!」がなくなった ← 影響が大きい
  • 痩せる等の目的がない ← 影響が大きい

2.1 課題の構造

行かない理由は「物理的な面倒」と「動機の枯れ」の二層に分かれる。

層 具体 アプリで効くか
物理的な面倒 遠い・歩きたくない・準備 部分的(準備の段取り化で軽減)
タイミング 夜だと帰りが遅い ◎ 自分に合う時間に行ける段取りで軽減
動機の枯れ 楽しくない・目的がない・忘れる ◎ 本アプリの主戦場

物理的な距離はアプリでは縮められない。**RepUp が狙うのは「動機の枯れ」と「タイミング」**である。タイミングについては、特定の時間帯を強制せず、ユーザーが選んだ時間で行ける段取りを支える(朝の扱いは §4.5)。

3. ゴール — To-Be(あるべき姿)

  • 筋トレに**「行かなきゃ」という気持ちになれる仕組み**がある
  • **続いている実感(積み重ね)**があり、継続的にジムに行けている

To-Be では時間帯(朝/夜)をゴールに含めない。 いつ行くかはユーザー個々の選択であり、アプリが特定の時間帯を強制しない。「朝に行く」は課題(特に夜のタイミングの難しさ)を解く**一手段(仮説)**であって、ゴールそのものではない。朝の扱いは §4.5 を参照。

4. GAP / HOW — 何で埋めるか

「行かなきゃの気持ちになれる仕組み」を、4つのメカニズムで作る。

4.1 段取りをアプリが持つ(忘れ・面倒への対策)

  • 行く日をあらかじめ設定し、当日・前日にリマインドを出す
  • 行く前日までに準備チェックリスト(持ち物・コワーキング予約など、カスタム可)が埋まっていなければアラート
  • 「忘れている」「準備がだるい」を、アプリ側の段取りで吸収する

4.2 報酬 — 行くことに意味を持たせる(動機の枯れへの対策)

行くたびに、複数の「積み重ね」が育つ。

  • 記録: 自分の成長具合が一目で見える(回数・継続)
  • 連続カウント: 「○回連続いけてる!💪」のような連続達成の可視化
  • 積み重ね: カウント・トロフィー(ブロンズ / シルバー / ゴールド)

4.3 ペナルティ — サボりに痛みを持たせる

  • サボると連続カウントが途切れる
  • トロフィーのランクが下がる(降格)
  • ペナルティは個人のトロフィー降格に閉じる(チームへの通知は持たない)

4.4 チーム — 一人にしない

  • チームを作る / 参加する
  • メンバーが行く日・進捗が見える
  • チーム内ランキングで互いに励む(MVP のランキングはチーム内のみ)

4.5 時間帯の提案(任意・仮説)

「いつ行くか」はユーザーが自由に選ぶ。そのうえで、ブレストで挙がった「朝に行く」利点を、強制せず任意の提案として提示できる。

  • 用事と被りにくい(夜は予定が入りやすい)
  • ジムが空いている(人が少ない)
  • 早起きが強制されることで生活リズムが整う

これは検証対象の仮説であって、ゴール(§3)ではない。アプリは時間帯を中立に扱い、朝は「選べる選択肢の一つとして軽く勧める」程度にとどめる(オンボーディングの既定提案・既定時間帯)。夜に行きたい人を妨げない。

5. 「行った証明」の扱い

MVP では 自己申告とする。

  • 記録が完了したら「行った」ことになる
  • 行っていない日はブランク状態として保持され、後から 「記録する」か「行かなかった」かを選んで確定できる

詳細な状態遷移は 02 機能設計 §F03 を参照。

6. ペルソナ

項目 内容
主役 ジム会費を払っているが足が遠のいている社会人。朝型にシフトしたい
状況 動機が枯れ、夜に行くのが負担。一人だと続かない
求めるもの 「行かなきゃ」と思える外圧と、続いている実感(積み重ね)
デバイス iPhone(iOS 優先)。通知を見る習慣はある

7. 機能スコープ

7.1 ベース機能

ID 機能 概要
BASE-01 認証・アカウント Sign in with Apple / メール+パスワード
BASE-02 プロフィール / オンボーディング 目的・行く頻度・通知の初期設定

7.2 MVP 機能

ID 機能 概要
F01 スケジュール & リマインド 行く日を設定、当日/前日にプッシュ。「いかなきゃ」の中核
F02 準備チェックリスト カスタム項目。前日に未達ならアラート
F03 記録(行った証明) 記録完了=行った。未記録日はブランク→後から確定
F04 報酬・積み重ね 連続カウント・トロフィー・成長グラフ
F05 ペナルティ サボりで連続途切れ + トロフィー降格
F06 チーム 作成/参加・進捗共有・チーム内ランキング

7.3 対象外(NA / 2nd 以降)

以下は MVP では実装しない。本節を対象外の唯一の根拠とし、他文書・他節では個別に「スコープ外」を散発的に書かない(必要なときは本節を参照する)。

項目 内容 見送り理由 / 移行時の論点
GPS チェックイン 位置で「行った」を自動判定 MVP は自己申告(§5)。位置許可・判定精度の検証が必要
全国ランキング / 「日本人の中で○位」 全ユーザー横断の順位 一定のユーザー母数が前提。MVP はチーム内ランキングのみ
育成マッチョ 筋トレで育つキャラクター 記録データに連動。まず習慣化ループの確立を優先
LINE / Google カレンダー連携 外部経路のリマインド MVP は iOS プッシュのみ。OAuth / 各 API 連携が必要
チームへの「晒し」ペナルティ 未達をチームに通知 MVP は個人のトロフィー降格のみ。関係性リスクを避ける

2nd の各項目は、それぞれ独立した spec → plan → 実装サイクルで進める(→ 07 ロードマップ §4)。

8. 成功の判断材料(MVP)

観点 見たいこと
継続(主) リマインドを受けて記録する人が、連続カウントを伸ばせているか
チーム効果 チームに属する人の記録頻度が、ソロより高いか
朝の提案(仮説/副次) 朝の提案を受け入れた人の継続が高いか。ゴールではなく検証対象(§4.5)

数値目標は MVP リリース後の実データで設定する。本書では「何を見るか」までを定義する。
