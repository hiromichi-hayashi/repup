DOCUMENT 02

機能設計書

バージョン: 1.0 作成日: 2026年6月 スコープ: BASE-01〜02 / F01〜F06 の機能仕様・API・状態遷移

本書は 01 コンセプト のスコープに従う。データ構造は 03 DB設計、画面は 04 画面設計 と相互参照する。

1. 全体像

1.1 システム構成

オンラインファースト。サーバーが真実のソース(SSoT)。端末はローカル DB を持たない。

[iOS / React Native + Expo]
   │  HTTPS (JSON / REST)
   ▼
[API: Hono (TypeScript)]
   │  Prisma
   ▼
[DB: Neon (PostgreSQL)]

[cron] ──▶ リマインド/アラート判定 ──▶ [Expo Push] ──▶ 端末

技術選定の根拠と棄却理由は 05 技術スタック、通知の配信設計は 06 通知設計 を参照。

1.2 API 規約

  • 認証は Bearer トークン(Better Auth セッション)。Authorization: Bearer <token>
  • 日時は 内部 UTC / 表示 JST。API は ISO8601(UTC)で送受信
  • スキーマは zod で定義し、リクエスト/レスポンスを検証
  • エラーは { "error": { "code": string, "message": string } } 形式、HTTP ステータスを併用

1.3 機能一覧

ID 機能 主なエンドポイント
BASE-01 認証 /auth/*(Better Auth)
BASE-02 アカウント / 目標設定 /account
F01 スケジュール&リマインド /schedules
F02 準備チェックリスト /checklist-templates, /checklists
F03 記録 /records
F04 報酬・ペナルティ /account/stats, /trophies
F06 チーム /teams, /teams/:id/members, /teams/:id/ranking

2. BASE-01 認証・アカウント

2.1 要件

  • ログインは次の 2 方式のいずれか(どちらで入っても同一アカウントに紐づく)
    • Sign in with Apple
    • メール + パスワード
  • メール+パスワードは、パスワードリセット(メールによる再設定フロー) とメール確認を含む
  • 認証基盤は Better Auth(Hono + Prisma 統合)。パスワードはハッシュ化して保存(平文保持しない)
  • 初回サインイン時に User レコードを作成し、初回設定(目標設定 / SC-02)へ誘導

2.2 エンドポイント

メソッド パス 説明
* /auth/* Better Auth が提供(Apple サインイン・メール+パスワードのサインアップ/サインイン・パスワードリセット・セッション)
GET /account 現在のユーザー情報 + 初回設定済みフラグ(詳細は §3)

3. BASE-02 アカウント / 目標設定

実体は アカウント情報 と 個人の目標設定 の2つ。いずれも個人単位で、チームは関与しない(チームは F06)。設定値は設定画面(SC-11)からいつでも編集できる。

3.1 アカウント情報

  • 表示名
  • メール(認証由来。BASE-01)
  • Push トークン(Expo Push、端末ごと)

3.2 目標設定(個人)

  • 目的 / 動機(任意, purpose): 「なぜ行くか」の自由記述。例「継続したい」
  • 目標回数(weeklyTarget): 週 N 回。「どれだけ行くか」の数値目標(リマインド・連続カウントの基準)
  • 既定の時間帯(defaultTimeOfDay): 朝 / 昼 / 夜(朝を既定として提案。強制しない。01 §4.5)

用語: 自由記述の「継続したい」は目的(purpose)、週 N 回が目標(weeklyTarget)。両者を区別する。

アカウント情報と目標設定は同一の /account リソースにまとめて扱う(目標設定を別エンドポイントに分けない)。

3.3 「オンボーディング」の位置づけ

「オンボーディング」は 独立した API ではなく、初回起動時に上記設定を一度通す UI フロー(画面 SC-02)の呼び名にすぎない。完了は初回の保存(PATCH /account)時にサーバーが onboardedAt を記録して表現する(専用の完了エンドポイントは持たない)。

3.4 エンドポイント

メソッド パス 説明
GET /account アカウント情報 + 目標設定 + 初回設定済みフラグ(onboardedAt)
PATCH /account アカウント情報・目標設定の更新(表示名・目的・週回数・既定時間帯)。初回保存時に onboardedAt を記録
POST /account/push-token Expo Push トークンを登録/更新

4. F01 スケジュール & リマインド

4.1 要件

「行く日」を事前に決め、当日(と前日)にプッシュで背中を押す。「いかなきゃ」を作る中核。

  • 単発予定(特定日)と、繰り返し(曜日指定の週次)の両対応
  • 予定ごとに時間帯(開始予定時刻)を持つ。朝をデフォルトに寄せる
  • 当日リマインド + 前日リマインド(チェックリスト未達アラートと連動、§F02)

4.2 データ

Schedule(詳細は 03 DB設計 §Schedule)。繰り返しは recurrenceRule(曜日の集合)で表現し、cron が当日分の対象を展開する。

4.3 エンドポイント

メソッド パス 説明
GET /schedules 自分の予定一覧(期間指定可)
POST /schedules 予定作成(単発 or 繰り返し)
PATCH /schedules/:id 予定変更
DELETE /schedules/:id 予定削除

4.4 リマインド配信

cron が定期実行し、対象予定を抽出して Expo Push を送る。配信タイミング・ペイロード・冪等性は 06 通知設計 に定義。

5. F02 準備チェックリスト

5.1 要件

行く前の段取りをアプリが持つ。「準備がだるい / 忘れる」への対策。

  • ユーザーはチェック項目テンプレートを自由に編集(例: タオル、シューズ、コワーキングスペース予約済み?)
  • 予定(Schedule)に紐づくチェックリストを生成し、各項目を ON/OFF
  • 前日時点で未達(未チェック項目あり)ならアラートを出す(§06)

5.2 データモデルの考え方

  • ChecklistTemplateItem: ユーザーごとの「項目の定義」(再利用される)
  • Checklist + ChecklistItem: 特定の予定に対する実体(チェック状態を持つ)

予定を作ると、テンプレートから当該予定用のチェックリストが複製される。テンプレートを後で変えても過去の予定の実体は変わらない。

5.3 エンドポイント

メソッド パス 説明
GET /checklist-templates テンプレート項目一覧
PUT /checklist-templates テンプレート項目の一括更新
GET /schedules/:id/checklist 予定のチェックリスト取得
PATCH /checklists/:id/items/:itemId 項目のチェック状態更新

6. F03 記録(行った証明)

6.1 要件

MVP の「行った証明」は自己申告。記録の完了をもって「行った」とする。

6.2 状態遷移

予定日(または任意の日)ごとに、その日の状態は次のいずれか。

            ┌──────────┐
            │  BLANK   │  予定日が来た / 任意日。まだ何も確定していない
            └────┬─────┘
       ┌─────────┴──────────┐
       ▼                    ▼
 「記録する」           「行かなかった」
       ▼                    ▼
 ┌──────────┐         ┌──────────┐
 │ RECORDED │         │  SKIPPED │
 │ =行った   │         │ =サボり   │
 └──────────┘         └──────────┘
  • BLANK: 確定前。予定があったが未操作の状態。後から RECORDED / SKIPPED のどちらかに確定できる
  • RECORDED: 記録(内容入力)を完了 → 「行った」。連続カウント・トロフィーに加算(§F04)
  • SKIPPED: 「行かなかった」を選択 → サボり。連続カウントを途切れさせ、降格評価の対象(§7.4)

BLANK のまま放置された過去日は「未確定」。連続カウントの判定では、確定するまで連続を伸ばさない(RECORDED で初めて加算)。降格の評価対象にするかは §7.4 で定義。

6.3 記録の内容(MVP)

  • 必須: 実施日
  • 任意: メモ、種目/部位の簡易タグ、所要時間。MVP では軽量に保ち、習慣化の可視化を優先

6.4 エンドポイント

メソッド パス 説明
GET /records 自分の記録一覧(期間指定可)
POST /records 記録を作成(= RECORDED に確定)
POST /records/skip 指定日を SKIPPED に確定
PATCH /records/:id 記録の編集
DELETE /records/:id 記録の取り消し(RECORDED → BLANK へ戻す)

7. F04 報酬・ペナルティ(積み重ねの増減)

記録(F03)に応じて積み重ねが増える(報酬)/減る(ペナルティ)。両者は同じ「積み重ね」システムの増減方向であり、ペナルティは独立機能(旧 F05)としては持たず本機能に内包する。

7.1 連続カウント(ストリーク)

RECORDED を 1 回ずつ積み上げる「来店の連続」。前回の来店から空きすぎる(許容ギャップ超過)か、SKIPPED を確定すると 0 に戻る。毎日行く前提は置かず、許容ギャップを「週 N 回」目標から導出して各自のペースに合わせる。

許容ギャップ(maxGapDays)

連続を維持できる「前回 RECORDED からの最大日数差」。ユーザーの weeklyTarget(週 N 回, 1〜7)から導出する。

maxGapDays = ceil(7 / weeklyTarget) + 1
weeklyTarget maxGapDays
週1 8
週2 5
週3 4
週4〜6 3
週7 2

gap は 2 つの RECORDED 日の日数差(例: 月→水 = 2)。gap ≤ maxGapDays なら連続を継続、超えると途切れる。+1 は「目標ペースより 1 日遅れても切れない」遊び。

算出(canonical)

連続は記録履歴から決まる導出値。UserStats.currentStreak / longestStreak にキャッシュし、Record の確定/取消時と日次 cron(放置による途切れ反映)で再計算する。評価日を today とする。

  1. 最新の確定日(RECORDED / SKIPPED のうち最も新しい日)が SKIPPED なら currentStreak = 0(SKIPPED 即リセット)
  2. 最新の RECORDED 日を Dlast とする。無ければ 0
  3. today − Dlast > maxGapDays なら、最新の連続は空きすぎで途切れている → currentStreak = 0
  4. それ以外は Dlast を起点に過去へ遡り、隣接する RECORDED の日数差が maxGapDays 以内である限り 1 ずつ加算。間に SKIPPED があればそこで打ち切る
  5. longestStreak = max(longestStreak, currentStreak)

1 日 1 確定(03 §Record の (userId, date) 一意)なので、同日重複は起きない。過去のブランク日を後から RECORDED にした場合も、本算出で連続が再計算される。

表示(SC-01)

  • currentStreak ≥ 1: 「○回連続いけてる!💪」
  • currentStreak = 0: 連続表示は出さず、「今日から始めよう」等の起点コピー

例(weeklyTarget = 3 → maxGapDays = 4)

  • 月・水・金に RECORDED → 連続 3(各 gap = 2 ≤ 4)
  • 月に RECORDED、その後 6 日空けて RECORDED → gap = 6 > 4 で途切れ、その回が連続 1
  • 月・水に RECORDED のあと木に SKIPPED 確定 → 連続 0(SKIPPED 即リセット)

7.2 トロフィー

累積記録数・連続記録に応じて ブロンズ / シルバー / ゴールド を付与する。

ランク 例(累積/連続の目安)
ブロンズ 初回〜継続の入口
シルバー 一定の累積 + 連続を達成
ゴールド 高い累積 + 長い連続を維持

付与の具体閾値は実装時に確定し、表を更新する。降格(ペナルティ)は §7.4。

7.3 成長グラフ

  • 記録回数の推移(週次/月次)を可視化し、「自分の成長具合が一目で見える」を満たす

7.4 ペナルティ(積み重ねの減少)

サボりに痛みを持たせる。ペナルティは個人のトロフィー降格に閉じる(チームへの通知は持たない / 対象外は 01 §7.3)。

  • 連続カウント途切れ: SKIPPED、または許容範囲を超えた空白で連続が 0 に戻る。F03 の SKIPPED 確定時に即時評価
  • トロフィー降格: サボりが続くと現在ランクが 1 段下がる(ゴールド→シルバー→ブロンズ)。cron による定期評価(一定期間記録がない / SKIPPED 連続)で判定。具体条件は実装時に確定し本節を更新

7.5 エンドポイント

メソッド パス 説明
GET /account/stats 連続カウント・累積回数・現在ランク・推移データ
GET /trophies 保有トロフィー一覧と次ランクまでの条件

8. F06 チーム

8.1 要件

一人にしない。チームで進捗を共有し、チーム内ランキングで励む。

  • チーム作成(作成者がオーナー)、招待コード or リンクで参加
  • メンバーの進捗が見える(直近の記録状況・連続カウント)
  • チーム内ランキング(連続カウント or 期間内記録数で順位付け)
  • ランキングはチーム内のみ(対象外は 01 §7.3)

8.2 プライバシー

  • 共有するのは記録の有無・回数・連続など進捗系のみ。記録メモ等の詳細は共有しない(MVP)

8.3 エンドポイント

メソッド パス 説明
POST /teams チーム作成
GET /teams/:id チーム情報・メンバー進捗
POST /teams/:id/join 招待コードで参加
DELETE /teams/:id/members/:userId 脱退/除名
GET /teams/:id/ranking チーム内ランキング(指標・期間指定)
