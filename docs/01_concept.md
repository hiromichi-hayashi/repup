# コンセプト / 要件定義

**バージョン**: 1.0
**作成日**: 2026年6月
**スコープ**: MVP のコンセプト・課題・ゴール・機能スコープの定義

> **RepUp** — 筋トレの「Rep(回数)」を積み「Up」する。行かなきゃの気持ちを仕組みで作り、朝ジム習慣へのシフトを支えるアプリ。

---

## 1. このドキュメントの位置づけ

本書は RepUp の **Single Source of Truth(SSoT)の起点**である。なぜ作るのか・誰の何を解決するのか・MVP に何を入れて何を入れないかを定義する。以降の機能設計(02)・DB設計(03)・画面設計(04)はすべて本書のスコープに従う。

実装と設計が食い違ったら、**コードを直す前に設計書を直す**。

---

## 2. 背景 — As-Is(現状の課題)

オーナー(初期ユーザー像)の生の声を起点にする。

- 筋トレに**お金を払っているのに行っていない**(ジム会費が無駄になっている)
- **夜に行く**のが暗黙のルールになっていて、行くと**帰りが遅くなる**のが嫌
- そもそも**行くのがめんどう**。ジムが遠い / 歩きたくない / 準備がだるい
- **時間がない(本当はなくはない)** という言い訳をしてしまう
- **もはや行くこと自体を忘れている**(習慣から外れている)
- **「筋トレ楽しい!」がなくなった** ← 影響が大きい
- **痩せる等の目的がない** ← 影響が大きい

### 2.1 課題の構造

行かない理由は「物理的な面倒」と「動機の枯れ」の二層に分かれる。

| 層 | 具体 | アプリで効くか |
|---|---|---|
| 物理的な面倒 | 遠い・歩きたくない・準備 | 限定的(物理的な距離は縮められない) |
| タイミング | 夜だと帰りが遅い | ◎ 自分に合う時間に行ける段取りで軽減 |
| 動機の枯れ | 楽しくない・目的がない・忘れる | ◎ 本アプリの主戦場 |

物理的な距離はアプリでは縮められない。**RepUp が狙うのは「動機の枯れ」と「タイミング」**である。タイミングについては、特定の時間帯を強制せず、ユーザーが選んだ時間で行ける段取りを支える(朝の扱いは §4.5)。

---

## 3. ゴール — To-Be(あるべき姿)

- 筋トレに**「行かなきゃ」という気持ちになれる仕組み**がある
- **続いている実感(積み重ね)**があり、**継続的にジムに行けている**

> **To-Be では時間帯(朝/夜)をゴールに含めない。** いつ行くかはユーザー個々の選択であり、アプリが特定の時間帯を強制しない。「朝に行く」は課題(特に夜のタイミングの難しさ)を解く**一手段(仮説)**であって、ゴールそのものではない。朝の扱いは §4.5 を参照。

---

## 4. GAP / HOW — 何で埋めるか

「行かなきゃの気持ちになれる仕組み」を、4つのメカニズムで作る。

### 4.1 リマインドで段取りを作る(忘れ・後回し対策)

- 行く日をあらかじめ**設定**し、当日・前日に**リマインド**を出す
- 「忘れている」「つい後回し」を、リマインドで防ぐ(物理的な距離は縮められないため、ここでは扱わない)

### 4.2 報酬 — 行くことに意味を持たせる(動機の枯れへの対策)

行くたびに、複数の「積み重ね」が育つ。

- **記録**: 自分の成長具合が一目で見える(回数・継続)
- **連続カウント**: 「○回連続いけてる!💪」のような連続達成の可視化
- **トロフィー(シーズン制ランク)**: 連続の最高到達で上がる6段階(ブロンズ→シルバー→ゴールド→プラチナ→ダイヤ→マスター)。ゲームのシーズンランクのように、**昇格のみ・降格なし**で、年ごとにリセット

### 4.3 ペナルティ — サボりに痛みを持たせる

- サボると**連続カウントが途切れる**(0 に戻る)— これがペナルティの中心
- **トロフィーの降格は行わない**。一度上げたランクはシーズン中は保持し、リセットは年次のみ(ゲームのシーズン制)

### 4.4 チーム — 一人にしない

- チームを作る / 参加する
- メンバーが行く日・進捗が見える
- **チーム内ランキング**で互いに励む(MVP のランキングはチーム内のみ)

### 4.5 時間帯の提案(任意・仮説)

「いつ行くか」はユーザーが自由に選ぶ。そのうえで、ブレストで挙がった「朝に行く」利点を、**強制せず任意の提案**として提示できる。

- **用事と被りにくい**(夜は予定が入りやすい)
- **ジムが空いている**(人が少ない)
- **早起きが強制される**ことで生活リズムが整う

これは検証対象の**仮説**であって、ゴール(§3)ではない。アプリは時間帯を中立に扱い、朝は「選べる選択肢の一つとして軽く勧める」程度にとどめる(オンボーディングの既定提案・既定時間帯)。夜に行きたい人を妨げない。

## 5. 「行った証明」の扱い

MVP では **自己申告**とする。

- **記録が完了したら「行った」ことになる**
- 行っていない日は**ブランク状態**として保持され、後から **「記録する」か「行かなかった」か**を選んで確定できる

詳細な状態遷移は [02 機能設計](02_function_design.html) §F03 を参照。

## 7. 機能スコープ

### 7.1 ベース機能

| ID | 機能 | 概要 |
|---|---|---|
| BASE-01 | 認証・アカウント | Sign in with Apple / メール+パスワード |
| BASE-02 | アカウント / 目標設定 | アカウント情報 + 個人の目標設定(目的・通う日・既定時間帯・通知許可) |

### 7.2 MVP 機能

| ID | 機能 | 概要 |
|---|---|---|
| F01 | スケジュール & リマインド | 行く日を設定、当日/前日にプッシュ。「いかなきゃ」の中核 |
| F03 | 記録(行った証明) | 記録完了=行った。未記録日はブランク→後から確定 |
| F04 | 報酬・ペナルティ | 連続カウント・トロフィー(シーズン制ランク・昇格のみ)・成長グラフ。サボりは連続リセット(降格なし)を内包 |
| F06 | チーム | 作成/参加・進捗共有・チーム内ランキング |

### 7.3 対象外(NA / 2nd 以降)

以下は **MVP では実装しない**。本節を対象外の**唯一の根拠**とし、他文書・他節では個別に「スコープ外」を散発的に書かない(必要なときは本節を参照する)。

| 項目 | 内容 | 見送り理由 / 移行時の論点 |
|---|---|---|
| GPS チェックイン | 位置で「行った」を自動判定 | MVP は自己申告(§5)。位置許可・判定精度の検証が必要 |
| 全国ランキング / 「日本人の中で○位」 | 全ユーザー横断の順位 | 一定のユーザー母数が前提。MVP はチーム内ランキングのみ |
| 育成マッチョ | 筋トレで育つキャラクター | 記録データに連動。まず習慣化ループの確立を優先 |
| LINE / Google カレンダー連携 | 外部経路のリマインド | MVP は iOS プッシュのみ。OAuth / 各 API 連携が必要 |
| チームへの「晒し」ペナルティ | 未達をチームに通知 | MVP のペナルティは個人の連続リセットに留める。関係性リスクを避ける |
| 準備チェックリスト | 行く前の持ち物・予約のチェック + 前日アラート | 使用頻度が低い見込み。コア(リマインド・記録・連続)を優先 |

> 2nd の各項目は、それぞれ独立した spec → plan → 実装サイクルで進める(→ [07 ロードマップ](07_implementation_roadmap.html) §4)。

## 8. 成功の判断材料(MVP)

| 観点 | 見たいこと |
|---|---|
| 継続 | リマインドを受けて記録する人が、連続カウントを伸ばせているか |
| チーム効果 | チームに属する人の記録頻度が、ソロより高いか |

> 数値目標は MVP リリース後の実データで設定する。本書では「何を見るか」までを定義する。
